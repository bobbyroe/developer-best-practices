Web Developer Best Practices

Introduction

If you want to learn more about building accessible web experiences, you’ve come to the right place.

The contents here are based on the W3C’s Web Content Accessibility Guidelines (WCAG 2.1). Our goal is to provide the simplest and most effective best practices on how to design, implement and test web applications with accessibility firmly in mind.

Why is digital accessibility important?

Inclusion is a core value here at T-Mobile, so we foster environment of inclusiveness for all customers and employees. About 25% of our customers have some form of disability (Visual, Auditory, Motor, Cognitive or Vestibular). Digital accessibility is the degree to which a website or application is useable to as many people as possible.

Accessibility defects are legal liabilities. T-Mobile websites and applications which are inaccessible open the company up to legal liabilities, from FCC complaints to ADA lawsuits, and can cost us millions.

T-Mobile will never become the innovative company we want to be if we continue to add accessibility defects to the backlog.

Setup

For creating web apps with accessibility in mind, you’ll need to get familiar with the basic concepts and have some specific tools at hand. Let’s go over how to get set up.

Udacity offers a superb, free course on Web Accessibility. Please watch at least the first 2 sections.

If you’re not running a Windows system yet, please work through the setup steps here: Window operating system setup. A Windows operating system machine is essential – most users with special needs have Windows-specific software running – like it or not.

The most widley used screen reader, currently is JAWS. You can download and install the trial version.

Lastly, visit the Chrome Web store to get the Lighthouse extension. Lighthouse adds automated accessibility (and other) scans to Chrome.

Invite an accessibility coach into the process

The best time to bring a coach into the process is as soon as it makes sense. The closer that is to the start of development, the better.

During planning, please reject stories that do not have defined accessibility acceptance criteria for UX/UI work. And be sure that your stories are sized with accessibility work included.

Knowledge

To begin developing accessible web apps, you need to understand how our customers use them, and how to optimize for their specific needs.

First thing we need to talk about is plain, traditional HTML. You may feel that being a serious senior full stack developer, this topic will have nothing new to offer. Turns out that most accessibility problems are the result of just bad HTML.

You’re probably accustomed to reading and using a web application on a traditional screen and control it with a mouse and keyboard. But, you may not realize that a surprisingly large group of customers rely on alternate devices to access the same information. So, get familiar with the fundamentals of those: how they work, how they are commonly used and what to pay attention to when providing support for them.

Before you read on, be sure to keep in mind that common knowledge of accessibility is full of misunderstandings, half-truths and technical oddities. This guide attempts to be fool proof and give the best answers to the most relevant questions.

  • Colors and contrast
    • Visual elements – text, icons, images and inputs need to be easily perceivable
  • Semantics and their importance for a11y
    • Learn how to use good markup
  • ARIA – when HTML is not enough
    • ARIA builds on HTML for use in web applications
  • Intro to keyboard only usage
    • Many customers cannot use a pointing device (like a mouse), so they rely on a keyboard. Thus your software must be fully functional with a keyboard only.
  • Intro to screen reader usage
    • Basics of interacting and reading websites and applications with a screen reader.

Keyboard accessibility is one of the most important aspects of web accessibility. As you develop and iterate, test your work using a keyboard to navigate though the user interface.
